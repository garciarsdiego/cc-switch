# 4.6 Cross-Protocol Routing

CC Switch can route selected Claude, Codex, and Gemini Native providers through the local proxy even when the client and upstream provider speak different API formats.

This feature requires the CC Switch local proxy and app takeover. Direct mode cannot perform protocol conversion because the CLI would call the upstream provider without CC Switch in the middle.

## Supported Directions

| Client app | Upstream provider format | Status |
|------------|--------------------------|--------|
| Claude Code | OpenAI Chat Completions | Supported through local routing |
| Claude Code | OpenAI Responses API | Supported through local routing |
| Claude Code | Gemini Native generateContent | Supported through local routing and Gemini API key or OAuth credentials |
| Codex | OpenAI Chat Completions | Supported through local routing |
| Codex | Anthropic Messages | Supported through local routing |
| Codex | Gemini Native generateContent | Supported through local routing and Gemini API key or OAuth credentials |

## Setup

1. Add or edit a provider for the target app.
2. For Codex, use one of the cross-protocol presets:
   - `Claude / Anthropic via Codex`
   - `Gemini Native OAuth/API key via Codex`
3. Paste the credential in the provider credential field. Gemini Native accepts an AI Studio API key, a bare `ya29` access token, or Gemini CLI `oauth_creds.json`.
4. Start the local proxy.
5. Enable takeover for the app that will send traffic through CC Switch.
6. Keep the proxy running while the CLI is active.

Providers that need conversion show a local-routing badge in the provider list. Codex Anthropic and Gemini Native presets also show a bridge warning in the provider form.

## Conversion Limits

Protocol conversion is best-effort. The proxy preserves common text, model, tool, and streaming flows, but some details are inherently lossy:

- Thinking and reasoning fields may be mapped to text-like deltas or provider-specific metadata.
- Tool calls are converted across schemas, but unusual tool argument shapes may differ by provider.
- Images can be stripped or normalized when the target model or protocol does not support the original image block.
- System prompts may be merged or relocated because providers place system instructions in different fields.
- Streaming events are normalized to the client format, so event names and chunk boundaries can differ from the upstream stream.

For production use, validate the exact model and prompt style you plan to run.

## Security Checklist

- Do not paste real credentials into screenshots, bug reports, fixture files, or test output.
- The proxy redacts Gemini OAuth tokens and client secrets in debug output.
- OAuth refreshes are serialized per refresh token to avoid overlapping refresh races.
- Provider credentials remain in the provider config path used by the selected app; local routing does not require adding credentials to global shell environment variables.
- Review provider terms before routing third-party traffic, especially for retained prompts, tool calls, or generated files.

## Live Validation Checklist

Run these checks with small prompts first. Stop the proxy after validation and confirm the app config is restored.

### Claude Code -> Codex/OpenAI-Compatible Provider

1. Add a Claude provider that uses OpenAI Chat Completions or OpenAI Responses API.
2. Start the proxy and enable Claude takeover.
3. Run a plain text Claude Code prompt.
4. Run a streaming prompt.
5. If the provider supports tools, run a simple tool-use prompt.
6. Confirm the provider card remains healthy and usage is recorded under the selected provider.

### Codex -> Claude/Anthropic Provider

1. Add the `Claude / Anthropic via Codex` preset.
2. Enter an Anthropic API key and keep the generated `apiFormat = anthropic` preset settings.
3. Start the proxy and enable Codex takeover.
4. Run a plain Codex prompt.
5. Run a streaming prompt.
6. Confirm Codex receives a Responses-compatible answer and the provider card shows the local-routing badge.

### Claude Code -> Gemini Native OAuth

1. Add or edit a Claude provider using Gemini Native format.
2. Paste an AI Studio API key, `ya29` access token, or Gemini CLI `oauth_creds.json`.
3. Start the proxy and enable Claude takeover.
4. Run a plain text prompt.
5. Run a streaming prompt.
6. If using OAuth JSON, repeat after token expiry or force a refresh path and confirm the refresh succeeds without exposing tokens in logs.

### Codex -> Gemini Native OAuth

1. Add the `Gemini Native OAuth/API key via Codex` preset.
2. Paste an AI Studio API key, `ya29` access token, or Gemini CLI `oauth_creds.json`.
3. Start the proxy and enable Codex takeover.
4. Run a plain Codex prompt.
5. Run a streaming prompt.
6. Confirm Codex receives a Responses-compatible answer and the provider card shows the local-routing badge.

## Local Smoke Checks Covered By Tests

- Codex Add Provider search finds the Anthropic and Gemini Native cross-protocol presets.
- Codex provider setup shows bridge warnings for Anthropic and Gemini Native presets.
- Codex provider cards show local-routing badges for Anthropic and Gemini Native bridges.
- The Claude adapter can read Codex-style `auth` and `config` fields when a Codex cross-protocol provider is routed through the Claude/Anthropic adapter.

## Known Risks

- Cross-protocol tool semantics vary by provider; validate high-value tool workflows before relying on them.
- Gemini OAuth depends on Google credential validity and refresh permissions.
- Provider-specific safety filters, rate limits, and billing behavior still apply after conversion.
- Long streaming responses are translated chunk by chunk, but upstream providers can still buffer internally before sending the first event.
